## Findings

### Minor Finding 1: Potential SQLite Open/Close Performance Overhead in Buffer Flush
- **What**: Opening and closing the SQLite database connection on every single deleted event during buffer flushing.
- **Where**: `edge-daemon/src/differential_privacy.cpp`, inside `deleteBufferedEvent()` and `flushBuffer()`.
- **Why**: When flushing up to 1000 events, the daemon opens and closes the SQLite database connection 1000 times, which is computationally expensive due to disk IO synchronization.
- **Suggestion**: In future optimization iterations, the system could wrap the deletion of multiple items in a single SQLite transaction, or keep the connection open for the duration of `flushBuffer()`. Since this is a background daemon task, the current performance impact is acceptable and does not impact functionality.

### Minor Finding 2: Lack of Negative Timestamp Check in verifySignature
- **What**: Missing validation for negative values in the `X-Timestamp` header.
- **Where**: `edge-daemon/src/differential_privacy.cpp`, line 878-900.
- **Why**: If an attacker crafts a highly negative timestamp string (e.g. `LLONG_MIN`), `std::stoll` parses it, and then evaluating `now_ms - request_time` triggers a signed overflow (equivalent to `now_ms + abs(LLONG_MIN)`), which is undefined behavior in C++.
- **Suggestion**: Explicitly check `if (request_time < 0) return false;` right after parsing the timestamp. (Note: standard API requests generated by the ChromeOS extension have positive timestamps, and the impact is mitigated as an invalid signature check would still fail verification).
